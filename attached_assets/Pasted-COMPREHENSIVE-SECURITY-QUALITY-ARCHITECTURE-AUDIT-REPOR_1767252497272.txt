1.1 HARDCODED JWT SECRET IN REPOSITORY
Severity: CRITICAL | Location: .replit:57

Issue:

JWT_SECRET = "uae-roots-prod-secure-jwt-secret-key-2024-x7k9m2p4q8w1"

The JWT secret is hardcoded in the .replit file, which is likely committed to version control. This means:

‚ùå The secret is publicly exposed if the repository is public
‚ùå Anyone with access can forge authentication tokens
‚ùå Complete authentication bypass is possible
Remediation:

IMMEDIATELY rotate the JWT_SECRET to a new cryptographically random value
Remove JWT_SECRET from .replit file
Add .replit to .gitignore if it contains secrets
Store JWT_SECRET only in environment variables or secure secret management
Audit all authentication logs for unauthorized access
Force re-authentication of all users
# Generate new secret (use this approach):
openssl rand -base64 48

1.2 DATABASE CONNECTION STRING LOGGED IN PLAINTEXT
Severity: CRITICAL | Location: server/db.js:16

Issue:

console.log(`Connecting to database at============== - ${process.env.DATABASE_URL}`);

The database connection string (containing credentials) is logged to console, which may be:

Written to application logs
Exposed in monitoring systems
Accessible to unauthorized personnel
Remediation:

// REMOVE this line entirely or redact credentials
const redactedUrl = process.env.DATABASE_URL?.replace(/:\/\/([^:]+):([^@]+)@/, '://***:***@');
console.log(`Connecting to database: ${redactedUrl}`);

1.3 ENCRYPTION KEY REUSE (JWT_SECRET)
Severity: HIGH | Location: server/index.js:61

Issue:

const ENCRYPTION_KEY = process.env.ENCRYPTION_KEY || JWT_SECRET;

The application falls back to using JWT_SECRET for PII encryption if ENCRYPTION_KEY is not set. This creates:

‚ùå Key reuse vulnerability - Same key for authentication and data encryption
‚ùå If JWT key is compromised, all PII is also compromised
‚ùå Violates cryptographic best practices (one key, one purpose)
Current State:

Warning is logged, but execution continues
In production, this should fail immediately
Remediation:

const ENCRYPTION_KEY = process.env.ENCRYPTION_KEY;
if (!ENCRYPTION_KEY) {
  console.error('CRITICAL: ENCRYPTION_KEY must be set separately from JWT_SECRET');
  process.exit(1);
}
if (ENCRYPTION_KEY === JWT_SECRET) {
  console.error('CRITICAL: ENCRYPTION_KEY and JWT_SECRET must be different');
  process.exit(1);
}

1.4 WEAK JWT EXPIRATION POLICY
Severity: HIGH | Location: server/index.js:1015, 1102

Issue:

jwt.sign({ userId: userId, type: "phone" }, JWT_SECRET, { expiresIn: "7d" });

Problems:

7-day JWT expiration is too long for a sensitive application
No refresh token mechanism
No token revocation capability
Stolen tokens remain valid for a week
Remediation:

Reduce JWT expiration to 1-2 hours maximum
Implement refresh tokens with 7-day expiration
Store refresh tokens in database for revocation capability
Add token rotation on refresh
1.5 INSUFFICIENT RATE LIMITING
Severity: MEDIUM-HIGH | Location: server/index.js:329-347

Current Limits:

API endpoints: 50 requests/minute per IP
SMS endpoints: 5 requests/15 minutes per IP
Issues:

No per-user rate limiting (only per-IP)
Distributed attacks can bypass IP-based limits
No progressive backoff for repeated failures
SMS limit is sufficient, but API limit may be too permissive
Remediation:

// Add user-based rate limiting
const userRateLimiter = rateLimit({
  windowMs: 1 * 60 * 1000,
  max: 30, // 30 requests per minute per user
  keyGenerator: (req) => req.userId || req.ip,
  skip: (req) => !req.userId, // Fall back to IP-based for unauthenticated
});

Complex credential retrieval logic increases attack surface
