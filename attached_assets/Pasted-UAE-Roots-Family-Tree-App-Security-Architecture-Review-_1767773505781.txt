UAE Roots Family Tree App - Security & Architecture Review
Executive Summary
This is a well-architected family genealogy application with strong security fundamentals. The codebase demonstrates mature security practices including PII encryption, comprehensive authentication, audit logging, and input validation. However, there are critical configuration vulnerabilities and architectural scalability concerns that need immediate attention.

Overall Assessment: 7.5/10

Security posture: Strong (with critical config gaps)
Architecture: Good (needs refactoring for scale)
Product completeness: Moderate (missing key features)
1. SECURITY & PRIVACY (Critical Issues Only)
🚨 CRITICAL VULNERABILITIES
C1: Firebase Credentials Exposed in Client Code
Severity: CRITICAL | Exploit Risk: HIGH

Issue: Firebase API keys and project configuration are embedded in client-side code via VITE_ environment variables (firebase.js:10-14), making them publicly accessible in the browser.

Impact:

Attackers can extract Firebase credentials from compiled JavaScript
Potential for API quota abuse, unauthorized Firebase service access
Firebase API key visible in network requests and source maps
Location: src/lib/firebase.js:10-14

Remediation:

// CURRENT (VULNERABLE):
apiKey: import.meta.env.VITE_FIREBASE_API_KEY, // Exposed in browser

// RECOMMENDED:
// 1. Move Firebase Admin SDK operations to backend only
// 2. Use Firebase App Check to validate client requests
// 3. Configure Firebase Security Rules to restrict access by domain
// 4. Enable API key restrictions in Google Cloud Console:
//    - HTTP referrers (web): *.uaeroots.com
//    - Application restrictions: Firebase only

Immediate Actions:

Go to Google Cloud Console → Credentials
Restrict API key to specific HTTP referrers (your domain)
Enable Firebase App Check in your Firebase project
Implement App Check verification in server/index.js
C2: Missing Environment Variable Validation
Severity: CRITICAL | Exploit Risk: MEDIUM

Issue: No .env.example file or comprehensive startup validation for required secrets.

Impact:

Production deployments may run with weak/default secrets
No validation that ENCRYPTION_KEY differs from JWT_SECRET (checked at line 76 but no enforcement if not set)
Twilio credentials failures only discovered at runtime
Location: server/index.js:39-81

Current State:

// Only JWT_SECRET is validated
if (!JWT_SECRET) {
  process.exit(1);
}
// ENCRYPTION_KEY falls back to JWT_SECRET (lines 61-73)

Remediation:
Create .env.example:

# Required secrets (must be different 32+ char random strings)
JWT_SECRET=your-jwt-secret-min-32-chars
ENCRYPTION_KEY=your-encryption-key-min-32-chars-different-from-jwt

# Database (required)
DATABASE_URL=postgresql://user:password@host:5432/dbname

# Firebase (required - get from Firebase Console)
VITE_FIREBASE_API_KEY=
VITE_FIREBASE_PROJECT_ID=
VITE_FIREBASE_APP_ID=

# Twilio SMS (required for phone auth)
TWILIO_VERIFY_SID=

# CORS (production)
ALLOWED_ORIGINS=https://uaeroots.com,https://www.uaeroots.com
NODE_ENV=production

Add startup validation in server/index.js:

// At startup, validate ALL required env vars
const requiredEnvVars = [
  'JWT_SECRET', 'ENCRYPTION_KEY', 'DATABASE_URL',
  'VITE_FIREBASE_PROJECT_ID', 'TWILIO_VERIFY_SID'
];

for (const varName of requiredEnvVars) {
  if (!process.env[varName]) {
    console.error(`CRITICAL: ${varName} environment variable is required`);
    process.exit(1);
  }
}

C3: Dual-Mode Decryption Vulnerability
Severity: HIGH | Exploit Risk: MEDIUM

Issue: Fallback decryption using legacy CryptoJS mode (line 121-122) could be exploited if attacker can inject malformed encrypted data.

Location: server/index.js:102-126

Problem:

// If v2 format fails, tries legacy CryptoJS (lines 121-122)
const bytes = CryptoJS.AES.decrypt(encrypted, ENCRYPTION_KEY);
return bytes.toString(CryptoJS.enc.Utf8);

Attack Vector:

Attacker crafts input that fails v2 validation but passes CryptoJS
CryptoJS uses different padding/mode than native crypto
Could lead to partial data exposure or DoS
Remediation:

Migration Strategy: Run a one-time migration script to convert ALL legacy CryptoJS encrypted data to v2 format
Remove Fallback: After migration, remove lines 121-122 entirely
Add Version Validation:
const decryptPII = (encrypted) => {
  if (!encrypted) return null;
  
  try {
    if (!encrypted.startsWith("v2:")) {
      // Log and reject legacy format after migration
      console.error("Rejected legacy encryption format");
      return "[DATA MIGRATION REQUIRED]";
    }
    
    const parts = encrypted.split(":");
    if (parts.length !== 4) {
      console.error("Invalid v2 encryption format");
      return null;
    }
    
    // ... existing v2 decryption logic
  } catch (error) {
    console.error("Decryption failed:", error.message);
    return null; // Don't expose encrypted data on error
  }
};

Migration Script: Check if server/migrate-encryption.js exists and run it before deploying this fix.

Severity: HIGH | Exploit Risk: MEDIUM

Severity: HIGH | Exploit Risk: LOW-MEDIUM

Issue: Resolved user ID stored in sessionStorage (api.js:209), accessible to XSS attacks.

Location: src/lib/api.js:200-214

Risk: If XSS vulnerability exists elsewhere, attacker can steal user ID and potentially craft requests.

Remediation:

// OPTION 1: Keep userId in memory only (lose on page refresh)
let authState = {
  resolvedUserId: null,
  timestamp: null,

// Remove sessionStorage.setItem() calls

// OPTION 2: Accept the risk but add integrity check
export function setAuthToken(token, userId) {

A7: No Caching Layer
Issue: Every request queries database, even for rarely-changing data (e.g., tree metadata, user profiles).

Impact:

Increased costs (Neon serverless charges per request)
Recommendation:
Implement Redis caching for:

Tree metadata (cache for 15 min)
People/relationships (invalidate on write)
import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);

// Wrapper for cached queries
async function cachedQuery(cacheKey, ttl, queryFn) {
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);
  
  const result = await queryFn();
  await redis.setex(cacheKey, ttl, JSON.stringify(result));
  return result;
}

// Usage
const people = await cachedQuery(
  `tree:${treeId}:people`,
  900, // 15 minutes
  () => db.select().from(people).where(eq(people.treeId, treeId))
);
